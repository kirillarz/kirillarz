# Журнал решений

## Назначение документа

Этот файл фиксирует важные продуктовые, технические и процессные решения проекта.

Агент должен читать его до начала работы над задачей.

Правила работы с решениями:

* агент может предложить новое решение или изменение существующего;
* агент не изменяет этот файл самостоятельно;
* агент не отменяет и не переопределяет принятое решение без явного подтверждения Кирилла;
* при неясности агент выбирает минимальный и обратимый вариант, сообщает о нём в отчёте, но не добавляет его в этот файл;
* решения из этого файла имеют приоритет над предположениями агента.

## D-001. Формат журнала решений

**Статус:** принято
**Дата:** 2026-06-24

### Контекст

В проекте будут использоваться ИИ-агенты. Важно не допустить, чтобы уже согласованные решения по продукту, дизайну, техническому стеку и процессу пересматривались без ведома владельца проекта.

### Решение

Использовать единый файл `docs/decisions.md` как журнал ключевых решений.

#### Шаблон решения

Каждое решение оформляется в формате:

```md
## D-XXX. Краткое название

**Статус:** принято | заменено | отменено  
**Дата:** YYYY-MM-DD

### Контекст
Почему возник вопрос.

### Решение
Что принято делать.

### Последствия и ограничения
Что это разрешает, запрещает или откладывает.

### Альтернативы
Какие варианты не выбраны.
```

### Последствия и ограничения

* Не создавать отдельную папку с множеством ADR-файлов на первом этапе.
* Не добавлять в журнал мелкие визуальные или текстовые правки.
* Записывать только решения, которые влияют на архитектуру, продукт, дизайн, зависимости, репозиторий, безопасность или процесс работы.

### Альтернативы

* Хранить отдельный ADR-файл для каждого решения.
* Не вести журнал решений.
* Описывать решения только в README.

## D-002. Агент не изменяет принятые решения

**Статус:** принято
**Дата:** 2026-06-24

### Контекст

ИИ-агент может обнаружить более удобный технический или визуальный вариант, но не должен самостоятельно менять продуктовые, архитектурные или процессные договорённости.

### Решение

Агент может предлагать изменения в `docs/decisions.md`, но не имеет права:

* самостоятельно редактировать этот файл;
* менять статус существующего решения;
* отменять принятое решение;
* добавлять новое стратегическое решение без явного подтверждения Кирилла.

### Последствия и ограничения

При неясной ситуации агент:

1. выбирает минимальный и обратимый вариант;
2. фиксирует предположение в итоговом отчёте;
3. предлагает отдельное решение для согласования;
4. не меняет `docs/decisions.md`.

### Альтернативы

* Разрешить агенту самостоятельно вносить технические решения.
* Требовать остановки работы при любой неоднозначности.

## D-003. Технический стек сайта

**Статус:** принято
**Дата:** 2026-06-24

### Контекст

Нужен современный стек для статического сайта-портфолио, который легко поддерживать, развивать и проверять через ИИ-агентов.

### Решение

Использовать:

* React;
* TypeScript;
* Vite;
* npm как пакетный менеджер;
* React Router для маршрутов `/` и `/employer`;
* CSS Modules для компонентных стилей;
* глобальные CSS-файлы для дизайн-токенов, базовой типографики, тем и общих правил.

### Последствия и ограничения

* Использовать `package-lock.json`.
* Все команды проекта должны быть совместимы с npm.
* Не использовать Tailwind CSS без отдельного решения.
* Не использовать UI-kit без отдельного согласования.
* Не добавлять state manager без реальной необходимости.
* Не добавлять отдельную библиотеку анимаций ради простых hover-эффектов и CSS-переходов.

### Альтернативы

* Vue или Next.js.
* pnpm.
* Tailwind CSS.
* Styled Components.
* CSS-in-JS.
* Отсутствие роутинга и одна страница без `/employer`.

## D-004. Сайт полностью статический

**Статус:** принято
**Дата:** 2026-06-24

### Контекст

Первая версия сайта — личное портфолио без пользовательских данных, личного кабинета и серверной логики.

### Решение

Первая версия сайта является полностью статическим frontend-приложением.

Не добавлять:

* backend;
* базу данных;
* CMS;
* авторизацию;
* API;
* серверную форму обратной связи;
* отправку писем с сайта;
* пользовательские аккаунты;
* сбор аналитики.

### Последствия и ограничения

* Контент хранится в коде и проектной документации.
* Связь с Кириллом происходит через Telegram.
* Сайт проще публиковать и поддерживать.
* При появлении backend-функций в будущем это должно быть отдельным архитектурным решением.

### Альтернативы

* Headless CMS.
* FastAPI backend.
* Серверная форма обратной связи.
* База данных для проектов и достижений.

## D-005. Документация — источник правды для контента

**Статус:** принято
**Дата:** 2026-06-24

### Контекст

Агент не должен придумывать факты о Кирилле, проектах, достижениях, технологиях или ссылках.

### Решение

Использовать следующий поток данных:

```text
docs/content.md
        ↓
src/data/portfolio.ts
        ↓
React-компоненты
```

Правила:

* факты сначала добавляются или исправляются в `docs/content.md`;
* затем они переносятся в структурированные данные сайта;
* React-компоненты не должны содержать вручную вшитые биографические тексты;
* агент не должен менять публичный контент без синхронизации с `docs/content.md`;
* при противоречии между текстом в компоненте и `docs/content.md` источником правды считается `docs/content.md`.

### Последствия и ограничения

* Описания проектов, контакты, технологии и достижения должны быть централизованы.
* Изменения контента легче проверять.
* Нельзя создавать отдельные противоречивые версии одного проекта в разных компонентах.

### Альтернативы

* Хранить весь контент прямо в JSX.
* Использовать CMS.
* Хранить контент только в README.

## D-006. Временные визуальные материалы допустимы

**Статус:** принято
**Дата:** 2026-06-24

### Контекст

Сайт нужно выпустить быстро, но финальные иллюстрации, обложки проектов, видео и скриншоты могут появиться позже.

### Решение

В первой версии допускаются временные визуальные материалы:

* CSS-placeholder-обложки;
* SVG-placeholder-обложки;
* временная пиксельная офисная сцена;
* нейтральные тематические композиции для карточек проектов.

Агенту разрешено создавать временные CSS/SVG-обложки самостоятельно.

### Последствия и ограничения

Временные материалы должны:

* выглядеть как намеренная часть интерфейса, а не как сломанная загрузка;
* соответствовать дизайн-системе;
* не использовать случайные стоковые изображения;
* не содержать ложной информации;
* быть легко заменяемыми финальными ассетами;
* иметь понятную структуру файлов и названий.

### Альтернативы

* Не публиковать сайт до готовности всех иллюстраций.
* Использовать случайные изображения из интернета.
* Оставлять пустые серые блоки вместо обложек.

## D-007. Генерация финальных изображений требует согласования

**Статус:** принято
**Дата:** 2026-06-24

### Контекст

Персонаж, пиксельный офис и финальные визуалы важны для личного бренда и не должны быть созданы случайно или без контроля художественного направления.

### Решение

Агент не генерирует финальные изображения без отдельной команды Кирилла.

Отдельное согласование требуется для:

* hero-иллюстрации персонажа;
* финального пиксельного офиса;
* финальных обложек проектов;
* ключевых иллюстраций;
* генерации изображений с использованием внешних моделей или сервисов.

Агент не должен брать изображения из интернета без явного разрешения и проверки лицензии.

### Последствия и ограничения

* До согласования разрешены только CSS/SVG placeholder-обложки.
* Генерация визуалов выполняется отдельной задачей.
* Финальные изображения хранятся как локальные ассеты проекта.
* При использовании чужих изображений необходимо отдельно проверить лицензию и источник.

### Альтернативы

* Разрешить агенту самостоятельно генерировать весь визуал.
* Использовать изображения из поиска без проверки лицензии.
* Не использовать иллюстрации вообще.

## D-008. Отсутствующие ссылки заменяются заглушками

**Статус:** принято
**Дата:** 2026-06-24

### Контекст

Не все ссылки на GitHub, видео, презентации, PDF-резюме и материалы проектов готовы на этапе MVP.

### Решение

Если ссылка отсутствует, интерфейс показывает аккуратную заглушку вместо активной ссылки.

Предпочтительный вариант:

* неактивная кнопка или тег;
* короткая подпись: «Скоро», «Материалы добавляются» или «В разработке»;
* отсутствие фиктивного URL;
* отсутствие перехода на пустую страницу или `#`.

### Последствия и ограничения

* Нельзя публиковать активную кнопку, которая ведёт на несуществующий ресурс.
* Заглушка не должна выглядеть как ошибка.
* После появления реальной ссылки агент может заменить заглушку на рабочее действие.
* Видимые пользователю `TODO` использовать нельзя.

### Альтернативы

* Полностью скрывать отсутствующее действие.
* Показывать текст без кнопки.
* Использовать временную фиктивную ссылку.

## D-009. Правила работы с Git

**Статус:** принято
**Дата:** 2026-06-24

### Контекст

Агенту разрешено выполнять часть Git-операций, но владелец проекта должен сохранять контроль над основной веткой и публикацией.

### Решение

Агенту разрешено:

* создавать локальные рабочие ветки;
* делать локальные коммиты;
* выполнять `git push` только в рабочие ветки;
* создавать pull request;
* обновлять собственный pull request.

Агенту запрещено:

* выполнять merge в `main`;
* делать force push;
* удалять ветки, теги или историю;
* пушить напрямую в `main`;
* менять branch protection rules;
* изменять права доступа к репозиторию;
* создавать релиз без отдельного согласования.

### Последствия и ограничения

* Все изменения должны попадать в `main` только после решения Кирилла.
* Ветки должны иметь понятные имена:

```text
feat/...
fix/...
docs/...
chore/...
```

* Коммиты должны быть небольшими и тематическими.
* Перед пушем агент обязан показать краткое описание изменений и результаты проверок.
* Pull request должен содержать краткое описание, список проверок и известные ограничения.

### Альтернативы

* Полностью запретить агенту Git-команды.
* Разрешить агенту выполнять merge.
* Работать без веток и pull request.

## D-010. Установка зависимостей ограничена

**Статус:** принято
**Дата:** 2026-06-24

### Контекст

Для простого статического портфолио лишние зависимости повышают сложность, риск уязвимостей и объём поддержки.

### Решение

Агент может установить dev-зависимость только после объяснения:

1. какую проблему она решает;
2. почему нельзя обойтись уже существующими средствами;
3. какие альтернативы рассматривались;
4. какая команда будет выполнена;
5. как зависимость будет использоваться и проверяться.

Production-зависимости нельзя добавлять без отдельного подтверждения Кирилла.

### Последствия и ограничения

* Разрешено запускать локальный dev server.
* Разрешено запускать `npm run build`, `npm run lint`, `npm run typecheck` и `npm run test`.
* На этапе пустого каркаса `lint`, `typecheck` и `test` могут временно быть явными заглушками, но перед публичным релизом должны быть заменены реальными проверками.
* Новая dev-зависимость должна быть зафиксирована в `package.json` и `package-lock.json`.
* Нельзя устанавливать зависимость только ради одной простой иконки, одной анимации или одной утилиты, которую легко написать самостоятельно.
* Агент не должен сам обновлять версии всех зависимостей «заодно».

### Альтернативы

* Полностью запретить установку зависимостей.
* Разрешить агенту свободно выбирать любые библиотеки.
* Использовать большой UI-kit и набор анимационных библиотек.

## D-011. Тестирование сфокусировано на полезной логике

**Статус:** принято
**Дата:** 2026-06-24

### Контекст

Сайт-портфолио содержит немного интерактивной логики. Полное покрытие тестами не является целью и может привести к бессмысленным тестам статичной разметки.

### Решение

Использовать Vitest для полезной логики:

* сохранение и восстановление темы;
* выбор системной темы при первом посещении;
* утилиты;
* трансформация данных;
* логика навигации или меню, если она вынесена в отдельные функции;
* другая интерактивная логика с риском регрессии.

Не писать тесты:

* только ради процента покрытия;
* для каждого статичного текста;
* для CSS-классов;
* в виде бессмысленных snapshot-тестов.

Playwright и автоматические visual regression tests не добавлять до первого релиза.

### Последствия и ограничения

* Качество визуала проверяется вручную на контрольных ширинах.
* После первого релиза можно отдельно принять решение о Playwright.
* При добавлении новой существенной интерактивной логики агент должен предложить тесты.

### Альтернативы

* Не использовать тесты.
* Ввести Playwright сразу.
* Использовать snapshot-тесты для всех компонентов.

## D-012. Агент не выполняет деплой

**Статус:** принято
**Дата:** 2026-06-24

### Контекст

Публикация сайта, подключение домена и доступ к хостингу требуют контроля владельца проекта.

### Решение

ИИ-агент только пишет, проверяет и подготавливает код.

ИИ-агенту запрещено:

* деплоить сайт;
* публиковать production-версию;
* подключать или менять домен `kirillarz.ru`;
* менять DNS-записи;
* заходить в панели хостинга;
* создавать или изменять секреты хостинга;
* создавать deployment workflow;
* добавлять автоматическую публикацию сайта;
* выполнять команды, которые публикуют сайт во внешнюю среду;
* создавать production release.

Деплой выполняет Кирилл вручную.

### Последствия и ограничения

Агент может:

* подготовить production build;
* проверить `npm run build`;
* подготовить пошаговую инструкцию по ручному деплою;
* проверить конфигурацию проекта на готовность к ручной публикации;
* указать, какие файлы необходимо загрузить или какие команды выполнить вручную.

Агент не должен выполнять инструкцию по деплою вместо Кирилла.

### Альтернативы

* Автоматический deploy через GitHub Actions.
* Деплой агентом через CLI.
* Деплой при каждом push в `main`.

## D-013. CI разрешён только для проверок качества

**Статус:** принято
**Дата:** 2026-06-24

### Контекст

Автоматическая проверка кода полезна, но CI не должен превращаться в механизм деплоя.

### Решение

Агенту разрешено создать обычный GitHub Actions workflow, который запускается при `push` и `pull_request` и выполняет только проверки качества.

Разрешённые шаги CI:

```text
npm ci
npm run lint
npm run typecheck
npm run test
npm run build
```

Если TypeScript, линтер или тесты ещё не настроены, соответствующие шаги могут быть временно пропущены или заменены явной заглушкой. Перед публичным релизом `lint`, `typecheck` и `build` должны быть реальными проверками.

### Последствия и ограничения

CI запрещено:

* публиковать сайт;
* подключаться к хостингу;
* использовать deployment secrets;
* менять DNS;
* создавать releases;
* запускать команды деплоя;
* загружать production-артефакты на внешний хостинг.

### Альтернативы

* Не использовать GitHub Actions.
* Использовать CI/CD с автоматическим деплоем.
* Запускать проверки только локально.

**Статус:** принято
**Дата:** 2026-06-24

Сайт публичный, а репозиторий может быть доступен другим людям. Нельзя публиковать лишние персональные данные, секреты и инструменты отслеживания.

### Решение

На сайте разрешено публиковать:

* имя: Кирилл Арзамасцев;
* GitHub;
* Telegram: `@kirillarz`;

На сайте и в репозитории запрещено публиковать:

## D-015. Поведение при неоднозначности

**Статус:** принято
**Дата:** 2026-06-24

### Контекст

При разработке могут возникать вопросы, которые не описаны в текущих документах: точный вид компонента, структура файлов, второстепенный UX-сценарий или техническая деталь.

### Решение

Если решение не определено в документации, агент:

1. выбирает минимальный и обратимый вариант;
2. не добавляет новую зависимость без объяснения;
3. не меняет публичный контент без источника;
4. не меняет `docs/decisions.md`;
5. сообщает о выбранном предположении в отчёте;
6. предлагает отдельный вариант для дальнейшего согласования, если вопрос влияет на продукт, дизайн, безопасность, деплой или архитектуру.

### Последствия и ограничения

* Работа не должна останавливаться из-за незначительных деталей.
* Агент не получает право принимать необратимые решения.
* Владелец проекта сохраняет контроль над стратегией сайта.

### Альтернативы

* Останавливать работу и задавать вопрос при любой неоднозначности.
* Разрешить агенту самостоятельно принимать все решения.

## D-016. Целевой домен и публикация

**Статус:** принято
**Дата:** 2026-06-24

### Контекст

У сайта будет собственный домен, но платформа хостинга ещё не выбрана.

### Решение

Целевой домен сайта:

```text
kirillarz.ru
```

Платформа хостинга выбирается отдельно.

До подключения собственного домена допустим временный технический адрес.

Деплой выполняется вручную Кириллом только из подготовленной и проверенной версии ветки `main`.

### Последствия и ограничения

* Агент не выбирает хостинг самостоятельно.
* Агент не настраивает домен.
* Агент не меняет DNS.
* Агент может подготовить инструкцию для ручной публикации после выбора хостинга.
* Решение о Vercel, GitHub Pages, Cloudflare Pages, Beget или другом варианте фиксируется отдельным пунктом в этом журнале.

### Альтернативы

* Выбрать платформу до начала разработки.
* Использовать только временный технический адрес.
* Использовать автоматическую публикацию из GitHub Actions.
