# Hero reference alignment plan

Этот документ — implementation brief для повторной работы над `portfolio-009`.
Пользователь не принял прежнее визуальное приближение: эталоном считается
`examples/hero-screen-desktop.jpg`, и следующая реализация должна быть
максимально близкой к нему, а не просто сохранять ту же общую композицию.

## Приоритет источников

1. Геометрия, масштаб, цвет, типографика и декоративные детали:
   `examples/hero-screen-desktop.jpg`.
2. Публичные факты и допустимые формулировки: `docs/content.md`.
3. Общая структура и ограничения проекта: `docs/site-brief.md`.
4. Текущая реализация — только baseline, а не источник визуальных решений.

Если CSS-значение из текущей реализации расходится с референсом, нужно менять
значение, а не пытаться сохранить его ради минимального diff.

## Scope

В scope одной фичи:

- статическая desktop-композиция hero;
- full-bleed темная сцена без видимой нижней полосы от page padding;
- позиция, масштаб и crop существующей минифигурки;
- позиция и ширина правой колонки;
- размер, насыщенность и перенос заголовка;
- перенос и межстрочные интервалы описания;
- четыре skill pills, точки-разделители и поведение ленты;
- размер, цвет, свечение, подпись и стрелка CTA;
- ненавязчивая подсказка под CTA и абстрактный световой знак справа, как в
  референсе;
- responsive-поведение, которое не ломает mobile viewport;
- reduced-motion fallback для ленты и любых декоративных микродвижений;
- visual smoke и ручное сравнение до/после.

Вне scope:

- сценарная transition-анимация со световым гаджетом из `portfolio-003`;
- новая навигация, мобильное меню и новые секции страницы;
- изменение маршрута CTA: он продолжает вести на `/employer`;
- генерация или замена `src/assets/hero-minifigure.png`;
- новые зависимости, web fonts и UI-библиотеки;
- изменение подтвержденного текста или списка направлений;
- unrelated refactor общих страниц и стилей.

## Canonical comparison setup

Desktop-референс содержит chrome браузера. Для сравнения самой страницы:

- исходный JPG: `1680 × 941`;
- верхние `103 px` относятся к браузеру и не входят в hero;
- эталонная область страницы: `1680 × 838`, начиная с `y = 103`;
- текущий сайт нужно снимать с viewport `1680 × 838`, без browser chrome и
  без full-page расширения кадра;
- дополнительно сохранить штатные smoke-кадры `1440 × 900` и `390 × 844`.

Основная приемка делается на `1680 × 838`. Значения ниже измерены по JPG
вручную и являются ориентирами с разумным допуском, а не pixel snapshot API.

## Что сейчас не совпадает

Baseline 2026-07-12 снят в `1680 × 838` и локально сохранен как
`artifacts/home-desktop-reference-viewport.png`.

| Зона | Эталон | Текущая версия | Что исправить |
|---|---|---|---|
| Нижний край | Фигура и фон доходят до низа viewport | После `y ≈ 790` видна пустая полоса около `48 px` | Убрать вертикальный page padding из full-bleed hero и клипать фигуру краем viewport |
| Фигура | Левый силуэт начинается около `x = 150`, правый около `x = 805` | Примерно `x = 190…870` | Сместить сцену влево и немного уточнить масштаб, не менять PNG |
| Правая колонка | Левый край около `x = 840` | Около `x = 828` | Сдвинуть колонку на `10–15 px` вправо через grid, а не случайный margin каждого элемента |
| Вертикаль контента | Заголовок начинается около `y = 145` | Около `y = 180` | Поднять композицию; учитывать будущую подсказку под CTA при центрировании всей группы |
| Заголовок | Две строки; правая граница второй строки около `x = 1455` | Вторая строка доходит примерно до `x = 1545` | Сделать начертание визуально уже/легче и подобрать размер по референсу |
| Описание | Две строки: перенос после `«продукты,»` | Перенос после `«объединяю»` | Ограничить measure и подобрать размер/line-height под точный перенос |
| Skills | Все четыре pills читаются в первом кадре | Последний pill обрезан | На широком desktop показывать полный статичный ряд; marquee нужен только когда ряд не помещается |
| CTA | Примерно `337 × 84 px` | Примерно `336 × 76 px` | Увеличить высоту, уточнить радиус, градиент, glow и расстояние до skills |
| Детали | Есть подсказка под CTA и световой ромб/блик справа | Оба элемента отсутствуют | Добавить легкие CSS/SVG-декорации без брендированного ассета |
| Фон | Более светлый и серо-синий, с мягкой неоднородностью | Правый фон почти черный, glow локальнее | Ослабить черный, добавить широкий сине-серый radial glow и легкую виньетку |

Контрольные цвета фона по свободным от объектов точкам эталонной page-area:

- левый верх: около `#070b14`;
- правый верх: около `#0a0e17` с уходом к `#050a10` у края;
- левый средний: около `#121a27`;
- нижняя зона: около `#030710…#040811`.

Не нужно добиваться этих hex-кодов одним плоским цветом: цель — похожее
распределение света и контраста.

## Целевая desktop-композиция

### Сцена и grid

- Hero занимает не меньше `100svh` и не получает видимый верхний/нижний inset
  от общего `.page`.
- Контент ограничен общей центральной сеткой, но фигуре разрешено выходить
  влево за ее колонку.
- Левая и правая части визуально уравновешены: центр лица находится примерно
  в левой трети, текст начинается около середины viewport.
- Crop выполняется рамкой сцены. Нельзя снова обрезать сам `<img>` так, чтобы
  при смене aspect ratio появлялись ноги или исчезали плечи.

### Минифигурка

- Использовать существующий прозрачный PNG; он уже соответствует персонажу
  референса и не требует повторной генерации.
- Верх волос — примерно `40–50 px` от верха page-area.
- Нижняя часть торса уходит за край viewport; отдельной темной полосы под
  изображением быть не должно.
- Мягкое голубое свечение должно читаться за головой и плечами, но не давать
  отдельный круг или карточку.

### Текст

- Заголовок остается `Кирилл Арзамасцев`, строго в две строки на canonical
  desktop.
- Не привязываться к текущим `6.5rem / 900`: подобрать комбинацию размера,
  weight, line-height и доступного системного font stack по визуальной ширине
  референса. Новую web-font зависимость не добавлять.
- Описание остается дословно:
  `Разрабатываю цифровые продукты, объединяю backend, AI и управление командой.`
- На `1680 × 838` первая строка описания должна заканчиваться словом
  `продукты,`, вторая начинаться с `объединяю`.

### Skills

- Порядок фиксирован: `Backend`, `AI Products`, `Project Management`,
  `Systems Analysis`.
- На canonical desktop все четыре pills и три разделителя видны одновременно.
- Начертание внутри pills ближе к regular/medium, а не к жирной кнопке.
- На широком desktop предпочтительно статичное состояние, совпадающее с
  референсом. Marquee включать только когда строка физически не помещается.
- При `prefers-reduced-motion: reduce` никакого автоскролла нет, весь список
  остается доступен визуально или через горизонтальный scroll.

### CTA и декор

- CTA сохраняет текст `Узнать обо мне`, стрелку вправо и маршрут `/employer`.
- Canonical размер — ориентировочно `337 × 84 px`; hit area не меньше `44 px`.
- Под кнопкой допускается референсная малоконтрастная подсказка
  `Нажми — будет эффектно` с короткой изогнутой стрелкой. Она не должна
  перехватывать focus или pointer events.
- Справа от CTA добавить абстрактный наклоненный контур и световой блик.
  Реализовать CSS или небольшим inline SVG; не использовать чужой ассет и не
  называть элемент кино-гаджетом.
- Декор скрывается раньше, чем начинает сталкиваться с текстом или CTA.

## Responsive strategy

Desktop-референс — главный критерий. Mobile должен быть аккуратной адаптацией,
но не расширяет scope до новой навигации:

- на `≤ 980 px` перейти к одной колонке: фигура сверху, текст ниже;
- на `390 × 844` не должно быть горизонтального overflow у страницы;
- имя остается в две строки и не касается краев;
- фигура не должна выглядеть уменьшенной и «висящей»: crop остается крупным,
  но между фигурой и заголовком нет лишней пустоты;
- skill row может быть горизонтальным scroll/marquee;
- CTA занимает доступную ширину;
- desktop-only подсказку и световой знак можно скрыть на узком экране;
- мобильная шапка `KA. / меню` из mobile-референса не добавляется в этой фиче:
  сейчас для нее нет подтвержденной navigation model. Если пользователь явно
  включит ее в scope, это решение нужно сначала записать в `docs/site-brief.md`.

## Рекомендуемый порядок реализации

1. Снять новый baseline в canonical viewport и не удалять предыдущий кадр до
   завершения сравнения.
2. В `HomePage.tsx` менять только структуру hero, необходимую для подсказки и
   декоративного знака. Не менять текст, route и semantic `h1`.
3. В `Page.module.css` сначала исправить full-bleed высоту, grid и crop. До
   настройки типографики добиться правильных границ фигуры и отсутствия
   нижней полосы.
4. Настроить правую колонку сверху вниз: heading, description, skills, CTA.
   Не компенсировать общий сдвиг разными magic margins на каждом дочернем
   элементе.
5. Сделать desktop skill row детерминированным; затем отдельно настроить
   overflow/marquee для узких viewport.
6. Добавить подсказку и световой знак последними, когда основная геометрия уже
   совпадает.
7. Проверить `980`, `640`, `390` и `320 px`, focus-visible и reduced motion.
8. Обновить visual smoke так, чтобы кроме текущих кадров было легко получить
   canonical `1680 × 838` screenshot. Pixel-diff test не обязателен.
9. Запустить полный verification и записать пути к новым кадрам в evidence.

## Известная проблема visual smoke в текущем окружении

На 2026-07-12 `npm run visual:smoke` не может запустить bundled Chromium:
Playwright ожидает `chromium_headless_shell-1228`, которого нет в локальном
cache. Это не дефект hero. Для локального анализа baseline был снят через уже
установленный системный Chrome. Следующий агент должен либо выполнить
`npx playwright install chromium` с разрешением на загрузку, либо временно
использовать системный Chrome для ручного screenshot; production-код ради
этого менять нельзя.
