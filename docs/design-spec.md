# Design Specification — личный сайт-портфолио Кирилла Арзамасцева

## 1. Назначение документа

Этот файл фиксирует визуальные и UX-решения для сайта-портфолио Кирилла Арзамасцева.

Все интерфейсные решения должны соответствовать этому документу. Если решение не описано явно, нужно выбирать вариант, который:

1. помогает посетителю быстрее понять позиционирование;
2. ведёт пользователя к разделу проектов;
3. сохраняет профессиональный характер сайта;
4. не перегружает интерфейс;
5. работает на мобильных устройствах;
6. не конфликтует с требованиями `docs/brief.md` и `docs/content.md`.

Сайт не должен выглядеть как шаблонное резюме, корпоративный лендинг без характера или полноценная браузерная игра.

---

## 2. Визуальное направление

### Основное впечатление

Сайт должен восприниматься как:

* технологичный;
* уверенный;
* энергичный;
* немного дерзкий;
* современный;
* собранный;
* живой, но не хаотичный.

### Чего нужно избегать

Сайт не должен выглядеть:

* детским;
* чрезмерно игровым;
* перегруженным анимациями;
* слишком «AI-шаблонным»;
* чрезмерно корпоративным и обезличенным;
* похожим на дашборд;
* похожим на лендинг с большим количеством градиентов и стеклянных карточек;
* как набор случайных карточек без визуальной иерархии.

### Референс

Сайт Trionn используется как ориентир по следующим принципам:

* крупная выразительная типографика;
* короткие уверенные формулировки;
* чистая композиция;
* сильный hero-блок;
* минимальное количество лишних UI-элементов;
* ощущение дизайнерского и технологичного продукта.

Не копировать:

* палитру;
* конкретные анимации;
* структуру;
* тексты;
* визуальные приёмы один в один.

## 3. Общая композиция сайта

### Структура основной страницы

```text
Главная / Hero
Обо мне
Проекты
Навыки
Опыт и сильные стороны
Достижения и обучение
Контакты
Footer
```

Отдельно должна существовать страница:

```text
/employer
```

### Страница `/employer`

Страница для работодателя должна быть отдельной, а не только секцией на главной.

Её задача — за 1–2 минуты показать:

* позиционирование Кирилла;
* желаемые направления;
* ключевые проекты;
* технические и управленческие навыки;
* лидерский и коммуникационный опыт;
* GitHub;
* PDF-резюме;
* Telegram для связи.

Страница должна быть более деловой и компактной, чем главная, но сохранять общую визуальную систему сайта.

## 4. Цветовая система

### Общий принцип

Основная палитра:

* глубокий тёмно-синий;
* электрический голубой;
* циан;
* нейтральные холодные поверхности;
* белый и почти белый текст;
* красный только как небольшой акцент в образе персонажа и отдельных редких деталях.

Красный галстук персонажа не должен становиться основным интерфейсным цветом.

### Тёмная тема

Тёмная тема должна напоминать аккуратный технологичный офис поздним вечером, а не киберпанк-игру.

```css
--color-bg: #07111f;
--color-surface: #0c1a2b;
--color-surface-raised: #10233a;
--color-surface-soft: #132946;

--color-text: #f4f8fc;
--color-text-muted: #a9b8ca;
--color-text-subtle: #7f90a6;

--color-border: #203650;
--color-border-strong: #315174;

--color-primary: #47c7ff;
--color-primary-hover: #7bd7ff;
--color-primary-active: #1caeea;

--color-accent: #00d9ff;
--color-accent-soft: rgba(0, 217, 255, 0.12);

--color-success: #47d7a8;
--color-warning: #ffbf69;
--color-danger: #ff6b6b;

--color-focus: #89eaff;
```

### Светлая тема

Светлая тема должна быть чистой, холодной и технологичной, без ощущения офисного шаблона.

```css
--color-bg: #f6f9fc;
--color-surface: #ffffff;
--color-surface-raised: #edf5ff;
--color-surface-soft: #e5f0fb;

--color-text: #0b1220;
--color-text-muted: #52657a;
--color-text-subtle: #708399;

--color-border: #d7e2ef;
--color-border-strong: #b9cbe0;

--color-primary: #006dff;
--color-primary-hover: #0057d9;
--color-primary-active: #0048b4;

--color-accent: #00a8c8;
--color-accent-soft: rgba(0, 168, 200, 0.1);

--color-success: #178f68;
--color-warning: #b66c00;
--color-danger: #cf3f3f;

--color-focus: #007fa6;
```

### Правила использования цветов

* Основной CTA использует `--color-primary`.
* Циан используется как визуальный акцент, фокус, небольшая декоративная линия, пиксельная деталь или активное состояние.
* Не использовать более одного яркого акцента в одном небольшом интерфейсном блоке.
* Цвет не должен быть единственным способом передать важную информацию.
* Текст поверх ярких акцентов должен проходить проверку на читаемость.
* Не использовать крупные кислотные градиенты.

## 5. Типографика

### Шрифтовая пара

Основной шрифт:

```text
Geist
```

Fallback:

```text
Inter, Arial, sans-serif
```

Моноширинный шрифт для технологий, тегов, статусов, мини-подписей и пасхалок:

```text
JetBrains Mono
```

Fallback:

```text
ui-monospace, SFMono-Regular, Consolas, monospace
```

### Правила типографики

* Основной текст — Geist.
* Моноширинный шрифт используется точечно.
* Не использовать моноширинный шрифт для длинных абзацев.
* Главные заголовки должны быть крупными, но не занимать весь экран на мобильных.
* Длина текстовых строк в абзацах должна оставаться комфортной для чтения.
* Не использовать больше двух гарнитур.

### Типографическая шкала

```css
--font-size-xs: 0.75rem;
--font-size-sm: 0.875rem;
--font-size-base: 1rem;
--font-size-lg: 1.125rem;
--font-size-xl: 1.25rem;
--font-size-2xl: 1.5rem;
--font-size-3xl: 2rem;
--font-size-4xl: 2.75rem;
--font-size-hero: clamp(2.6rem, 5vw, 4.75rem);
```

### Иерархия

* Hero H1: `--font-size-hero`.
* Заголовок секции: от `2rem` до `2.75rem`.
* Заголовок карточки проекта: от `1.25rem` до `1.5rem`.
* Основной текст: `1rem` или `1.125rem`.
* Теги и подписи: `0.75rem`–`0.875rem`.
* Межстрочный интервал основного текста: от `1.5` до `1.7`.
* Межстрочный интервал заголовков: от `1.05` до `1.2`.

```css
--container-padding-mobile: 20px;
--container-padding-tablet: 32px;
--container-padding-desktop: 48px;
```

### Шкала отступов

Использовать только значения из шкалы:

```css
--space-1: 4px;
--space-2: 8px;
--space-3: 12px;
--space-4: 16px;
--space-5: 20px;
--space-6: 24px;
--space-8: 32px;
--space-10: 40px;
--space-12: 48px;
--space-16: 64px;
--space-20: 80px;
```

### Радиусы

```css
--radius-sm: 12px;
--radius-md: 18px;
--radius-lg: 24px;
--radius-xl: 32px;
```

### Тени

Тени должны быть мягкими и почти незаметными.

В тёмной теме тени должны быть более прозрачными, а визуальное разделение поверхностей должно достигаться в первую очередь разницей фона и границ.

## 7. Адаптивные состояния

### Контрольные ширины

Обязательная проверка интерфейса:

* 390 px — мобильное устройство;
* 768 px — планшет;
* 1024 px — небольшой ноутбук;
* 1440 px — широкий десктоп.

### Breakpoints

```css
--breakpoint-tablet: 768px;
--breakpoint-laptop: 1024px;
--breakpoint-desktop: 1280px;
--breakpoint-wide: 1440px;
```

### Mobile-first правила

* Интерфейс проектируется сначала для мобильной ширины.
* На мобильном все основные блоки находятся в один столбец.
* Карточки проектов на мобильном всегда идут в один столбец.
* Не должно быть горизонтального скролла.
* Текст не должен быть слишком мелким.
* Навигация не должна перекрывать контент.
* Hero-иллюстрация упрощается до компактной композиции.
* Декоративный пиксельный офис не должен занимать весь первый экран на мобильном.

## 8. Навигация

### Desktop

Навигация находится в верхней части страницы и остаётся липкой при прокрутке.

Состав:

```text
Логотип / имя
Обо мне
Проекты
Навыки
Опыт
Достижения
Контакты
Переключатель темы
```

Правила:

* шапка не должна быть высокой;
* при прокрутке она получает лёгкий фон и тонкую границу;
* активный раздел может подсвечиваться;
* навигация должна оставаться спокойной и не конкурировать с hero-блоком;
* ссылка на `/employer` может быть отдельным компактным текстовым действием или пунктом меню.

### Mobile

На мобильном используется обычное раскрывающееся меню.

Правила:

* меню открывается по кнопке;
* кнопка имеет понятное доступное название;
* меню закрывается по `Esc`;
* меню закрывается после перехода к разделу;
* пункты меню имеют крупную зону нажатия;
* меню не должно быть тяжёлым полноэкранным игровым интерфейсом;
* переключатель темы доступен внутри мобильного меню.

## 9. Hero-блок

### Композиция

На десктопе:

```text
Текст и CTA слева
Иллюстрация персонажа и декоративный пиксельный офис справа
```

На мобильном:

```text
Текст
CTA
Компактная иллюстрация
```

### Содержание hero-блока

Обязательные элементы:

```text
Кирилл Арзамасцев
Начинающий product manager и разработчик
Собираю AI-продукты, команды и работающие решения.
[ Смотреть проекты ]
```

### Ограничения

* Не добавлять над именем дополнительный eyebrow label.
* Не добавлять блоки с цифрами, статистикой и «3 проекта / 6 человек / 99/100» в hero.
* Использовать только одну основную CTA-кнопку.
* Основная CTA: `Смотреть проекты`.
* Не добавлять вторую крупную CTA-кнопку.
* GitHub и резюме доступны в навигации, контактах и других разделах сайта.

### Визуальная логика

Hero должен быть визуально сильным, но текст остаётся главным.

Порядок восприятия:

1. имя;
2. роль;
3. короткая фраза;
4. CTA;
5. персонаж и офис как усиление образа.

## 10. Иллюстрация персонажа

### Стиль

Основной персонаж — современная стилизованная иллюстрация, а не реалистичный портрет и не пиксельный спрайт.

Персонаж должен выглядеть как уверенный PM и энергичный организатор.

### Одежда

* синий костюм-двойка;
* синий пиджак;
* синие брюки;
* белая рубашка;
* красный галстук;
* тёмные очки.

### Ограничения

* Не пытаться добиваться точного портретного сходства.
* Не добавлять рядом крупные предметы-символы вроде ноутбука, канбан-доски, кружки, AI-иконки или стопки стикеров.
* Персонаж должен быть узнаваемым как лидер / PM, но без карикатурного «босс-вайба».
* Не делать персонажа слишком серьёзным или агрессивным.
* Не использовать фотореализм.

---

## 11. Пиксельный офис

### Роль

Пиксельный офис — декоративный вау-элемент на главной странице.

Он не должен быть игрой, интерактивной картой или заменой навигации.

### Визуальный стиль

* классический pixel art;
* аккуратный, читаемый на небольшом размере;
* связан с атмосферой PM Simulator;
* сочетается с сине-циановой палитрой;
* имеет немного ироничный характер;
* остаётся вторичным относительно текста hero-блока.

### Обязательные зоны

В сцене должны присутствовать:

* канбан-доска;
* доска с идеями;
* кухня или кофемашина;
* рабочая атмосфера IT-команды перед релизом.

### Анимации

Допускаются только мелкие ненавязчивые циклические анимации:

* лёгкое мигание экрана;
* небольшое изменение содержимого монитора;
* движение курсора;
* пар от кофемашины;
* редкое изменение статуса на канбан-доске;
* мягкое свечение одного акцентного элемента.

### Запреты

* не использовать параллакс;
* не использовать управление персонажем;
* не делать объекты кликабельными;
* не использовать шумные эффекты;
* не запускать несколько заметных анимаций одновременно;
* не делать офис полноценной игрой;
* не использовать офис как фон под основной текст.

### Сочетание с современной иллюстрацией

Современная иллюстрация персонажа и пиксельный офис должны быть визуально разделены.

Допустимые варианты:

* персонаж находится в основной части hero-блока, офис — в отдельной декоративной карточке;
* персонаж расположен рядом с офисом, но не является частью пиксельной сцены;
* офис находится на заднем плане как отдельный фрагмент интерфейса.

Недопустимый вариант:

* современный персонаж буквально встроен в пиксельную сцену как объект другого художественного стиля.

## 12. Блок «Обо мне»

### Роль блока

Блок «Обо мне» — главный визуальный акцент после hero.

Он должен быстро объяснить, чем Кирилл отличается от обычного начинающего разработчика или студента.

### Композиция

На десктопе:

```text
Слева — короткий сильный текст
Справа — 2–3 смысловых тезиса или компактные карточки
```

На мобильном:

```text
Текст
Тезисы / карточки
```

### Смысловые акценты

Использовать три направления:

```text
Разработка
Управление
Организация
```

Пример логики блока:

```text
Идея → команда → работающий результат
```

### Ограничения

* Не делать длинную биографию.
* Не перечислять все достижения.
* Не повторять полный список технологий.
* Не превращать блок в резюме.
* Не использовать больше трёх небольших карточек или тезисов.

---

## 13. Карточки проектов

### Общий стиль

Карточки проектов должны быть дружелюбными и современными:

* крупные скругления;
* лёгкие тени;
* чистые поверхности;
* понятная иерархия;
* тематические обложки;
* короткие описания;
* спокойные технологические теги.

### Визуальный контент карточек

Для первой версии вместо реальных скриншотов использовать:

* пиксельные иллюстрации;
* тематические обложки;
* минимальные абстрактные визуальные композиции;
* символические элементы, связанные с проектом.

Не использовать пустые шаблонные изображения.

### Тематические направления

#### AI-агент для подбора помещений

Визуальная тема:

* карта;
* связанные точки;
* карточки объявлений;
* AI-слой;
* сине-циановая технологичная композиция.

#### BotNetSchool

Визуальная тема:

* Telegram;
* VK;
* календарь;
* расписание;
* уведомления;
* бот как проводник между сервисами.

#### PM Simulator

Визуальная тема:

* пиксельный офис;
* канбан;
* команда;
* дедлайн;
* релиз;
* игровой, но профессиональный характер.

### Содержание карточки

Каждая карточка должна включать:

* тематическую обложку;
* название;
* короткое описание;
* роль;
* ключевые теги;
* одну или две ссылки.

### Теги

Метрики и факты показывать спокойно в виде тегов.

Примеры:

```text
PM + Frontend
Сбер
6 человек
99/100
Telegram + VK
MVP
Python
React
FastAPI
```

Не делать метрики огромными цифрами.

### Hover-состояние

На десктопе карточка может:

* слегка подниматься;
* получать более заметную тень;
* подсвечивать границу;
* делать изображение немного ярче.

Hover не должен быть единственным способом увидеть важную информацию.

## 14. Навыки, опыт и достижения

### Навыки

Навыки показывать группами:

```text
Разработка
Продукт и управление
Коммуникация и организация
```

Визуальный стиль:

* компактные теги;
* моноширинный шрифт;
* спокойный фон;
* без шкал и рейтингов;
* без процентов;
* без прогресс-баров.

### Опыт и сильные стороны

Использовать более текстовый и человеческий блок.

Визуальная логика:

```text
Координирую
Объясняю
Защищаю результат
```

Можно использовать небольшие обычные line icons.

### Достижения и обучение

Использовать компактную временную линию или список.

Показывать:

* год;
* название;
* результат;
* краткий контекст.

Не использовать огромные медали, кубки или золотые 3D-иконки.

---

## 15. Иконки и декоративные элементы

### Иконки

Использовать смешанный подход:

* обычные минималистичные line icons — для интерфейса;
* пиксельные иконки — только в пиксельном офисе и отдельных декоративных деталях.

### Допустимые декоративные элементы

* тонкая сетка;
* лёгкие координатные подписи;
* редкие моноширинные технические статусы;
* аккуратные курсоры;
* небольшие интерфейсные стикеры;
* отдельные фрагменты кода;
* тонкие линии и маркеры.

### Ограничения

* Декоративные элементы не должны мешать тексту.
* Не использовать терминальные окна как большие фоновые блоки.
* Не использовать искусственный «хакерский» стиль.
* Не делать каждую секцию технической.
* Не использовать декоративные элементы одновременно во всех блоках.

---

## 16. Анимации и взаимодействия

### Общий уровень

Допускаются лёгкие анимации.

### Допустимые анимации

* мягкое появление секций при прокрутке;
* небольшое смещение элементов;
* hover-состояния кнопок;
* hover-состояния карточек;
* плавное переключение темы;
* аккуратное открытие мобильного меню;
* микро-анимации пиксельного офиса.

### Тайминги

```css
--motion-fast: 120ms;
--motion-base: 180ms;
--motion-slow: 260ms;
```

### Правила

* Анимации должны быть короткими.
* Основные переходы не должны быть длиннее 260ms.
* Не использовать bounce-анимации.
* Не использовать бесконечные заметные движения.
* Не использовать анимации, которые мешают чтению.
* Не использовать анимации только ради эффекта «дорогого сайта».

### Reduced motion

При `prefers-reduced-motion: reduce`:

* отключить появление элементов при скролле;
* отключить смещения и масштабирование;
* отключить циклические анимации пиксельного офиса;
* оставить только мгновенные или почти мгновенные изменения состояния;
* сохранить понятные hover и focus-состояния без движения.

## 17. Доступность

### Обязательные требования

* Использовать семантическую HTML-разметку.
* Сохранять логический порядок заголовков.
* Делать все интерактивные элементы доступными с клавиатуры.
* Добавить отчётливый видимый focus-state.
* Не скрывать focus-state.
* Не делать цвет единственным способом показать статус.
* Давать альтернативный текст содержательным изображениям.
* Помечать декоративные изображения как декоративные.
* Обеспечить читаемый контраст в светлой и тёмной темах.
* Поддерживать `prefers-reduced-motion`.
* Делать области нажатия удобными для мобильных пользователей.
* Не прятать важную информацию только в hover-состоянии.
* Липкая шапка не должна перекрывать целевой блок при навигации по якорной ссылке.

### Focus-state

Focus-state должен быть заметным в обеих темах.

Рекомендуемый стиль:

```css
outline: 3px solid var(--color-focus);
outline-offset: 3px;
```

### Размеры интерактивных элементов

* Основные кнопки: минимум 44×44 px.
* Иконки-кнопки: минимум 40×40 px.
* Ссылки в тексте должны иметь заметное состояние при hover и focus.
* Пункты мобильного меню должны иметь крупную область нажатия.

## 20. Проверка готового интерфейса

Перед завершением реализации необходимо проверить:

### Визуально

* Hero понятно читается за несколько секунд.
* Главный CTA заметен.
* Текст не проигрывает иллюстрации.
* Пиксельный офис выглядит как фишка, а не как игра.
* Блок «Обо мне» заметен сразу после hero.
* Карточки проектов выглядят единым набором.
* Светлая и тёмная темы сохраняют характер сайта.
* Не используется больше одного сильного акцента в одном блоке.

### На адаптиве

* Проверить 390 px.
* Проверить 768 px.
* Проверить 1024 px.
* Проверить 1440 px.
* Нет горизонтального скролла.
* Карточки проектов на мобильном в один столбец.
* Hero-иллюстрация на мобильном упрощена.
* Мобильное меню открывается и закрывается корректно.

### По доступности

* Все ссылки и кнопки доступны с клавиатуры.
* Focus-state заметен.
* Переключатель темы доступен с клавиатуры.
* Анимации отключаются при reduced motion.
* Декоративные элементы не мешают чтению.
* Содержательные изображения имеют alt-текст.
* Основные кнопки достаточно крупные для нажатия.

### По качеству реализации

* Не использовать жёстко зашитые цвета вне дизайн-токенов.
* Не использовать случайные значения отступов вне шкалы.
* Не добавлять новую визуальную систему для одной карточки или секции.
* Не добавлять зависимости только ради простой анимации или иконки.
* Не менять дизайн-направление без обновления этого документа.
