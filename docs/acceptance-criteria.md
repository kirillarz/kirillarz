# Acceptance Criteria — личный сайт-портфолио Кирилла Арзамасцева

## 1. Назначение документа

Этот файл определяет условия, при которых версия сайта считается готовой к публикации или принятию.

Критерии применяются вместе с:

* `docs/brief.md`;
* `docs/content.md`;
* `docs/design-spec.md`;
* `AGENTS.md`.

Если реализация конфликтует с этими документами, приоритет имеют:

1. достоверность контента;
2. работоспособность пользовательских сценариев;
3. читаемость и доступность;
4. визуальное соответствие дизайн-спецификации;
5. дополнительные декоративные улучшения.

---

## 2. Цель первого релиза

Первая версия сайта должна быть сильным, работающим и публично доступным MVP-портфолио.

Приоритеты первого релиза:

1. Понятное позиционирование Кирилла.
2. Быстрый переход к проектам.
3. Корректная работа главной страницы и `/employer`.
4. Светлая и тёмная темы.
5. Адаптивность.
6. Достоверный контент.
7. Возможность постепенно заменить временные иллюстрации, ссылки и ассеты на финальные.

Первая версия не обязана содержать:

* финальные иллюстрации всех проектов;
* реальные ссылки на все репозитории, видео и презентации;
* PDF-резюме по активной ссылке;
* все готовые медиа-материалы;
* автоматические visual regression tests;
* аналитику;
* SEO-настройки, Open Graph, favicon, sitemap и `robots.txt`.

---

## 3. Definition of Done для любой завершённой задачи

Задача считается завершённой только если:

* её результат соответствует утверждённому плану;
* не изменены несвязанные файлы без причины;
* нет выдуманных фактов о Кирилле, проектах, технологиях или достижениях;
* код проходит линтер;
* TypeScript-проверка проходит;
* production-сборка проходит;
* изменения просмотрены в браузере, если задача влияет на интерфейс;
* в отчёте указаны изменённые файлы, команды проверки и известные ограничения;
* новые `TODO` не маскируются под завершённую функциональность.

Обязательные команды перед завершением значимой задачи после настройки соответствующих инструментов:

```bash
npm run lint
npm run typecheck
npm run build
```

Если в проекте уже есть тесты или добавляется интерактивное поведение, дополнительно запускать:

```bash
npm run test
```

Если `lint`, `typecheck` или `test` временно являются заглушками, это нужно явно указать в отчёте. Публичный релиз нельзя считать готовым, пока обязательные проверки остаются заглушками.

## 5. Обязательные пользовательские сценарии

### Сценарий 1. Переход к проектам

**Дано:** пользователь находится на первом экране главной страницы.
**Когда:** пользователь нажимает «Смотреть проекты».
**Тогда:** страница плавно или мгновенно переводит пользователя к разделу проектов.
**И:** заголовок раздела не скрывается под липкой шапкой.

### Сценарий 2. Переключение темы

**Дано:** пользователь находится на любой странице сайта.
**Когда:** пользователь меняет светлую тему на тёмную или наоборот.
**Тогда:** интерфейс полностью переключается без нечитаемого текста и сломанных поверхностей.
**И:** после обновления страницы выбранная тема сохраняется.
**И:** при первом посещении, если сохранённого выбора нет, используется системная тема пользователя.

### Сценарий 3. Страница для работодателя

**Дано:** пользователь находится на главной странице или открывает `/employer` напрямую.
**Когда:** пользователь переходит на `/employer`.
**Тогда:** он быстро находит позиционирование, ключевые проекты, направления развития и доступные способы связи.
**И:** страница работает после прямого перехода и обновления браузера.

### Сценарий 4. Мобильное меню

**Дано:** пользователь использует мобильный экран.
**Когда:** пользователь открывает меню.
**Тогда:** он видит доступные разделы, переключатель темы и переход на `/employer`.
**Когда:** пользователь выбирает раздел или нажимает `Esc`.
**Тогда:** меню закрывается.
**И:** меню не перекрывает контент после перехода.

Проверка обязательна на ширинах:

* 390 px;
* 768 px;
* 1024 px;
* 1440 px.

| Критерий                                                                          | Проверка                    |
| --------------------------------------------------------------------------------- | --------------------------- |
| На 390 px отсутствует горизонтальный скролл.                                      | Проверить страницу целиком. |
| На 390 px навигация открывается через обычное раскрывающееся меню.                | Проверить меню.             |
| На 390 px карточки проектов идут в один столбец.                                  | Проверить раздел проектов.  |

| На 1024 px и 1440 px hero помещается в один экран вместе с CTA.                   | Визуальная проверка.        |

| Липкая навигация не закрывает заголовки разделов при переходе по якорным ссылкам. | Проверить переходы в меню.  |

## 8. Доступность

Доступность проверяется вручную и по возможности автоматизированными инструментами. Автоматические проверки полезны, но не заменяют ручной проход с клавиатурой. WCAG 2.2 отдельно подчёркивает видимость фокуса, логичный порядок фокуса, отсутствие перекрытия сфокусированного элемента и достаточный размер интерактивных целей.

| Критерий                                                                                                     | Проверка                                                            |
| ------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------- |
| Весь основной сайт можно пройти только клавиатурой.                                                          | Пройти сайт клавишей `Tab`, `Shift + Tab`, `Enter`, `Space`, `Esc`. |
| Фокус виден на ссылках, кнопках, меню и переключателе темы.                                                  | Проверить клавиатурой в обеих темах.                                |
| Порядок фокуса соответствует визуальному и смысловому порядку страницы.                                      | Пройти страницу с клавиатуры.                                       |
| Липкая шапка не перекрывает элемент, получивший фокус.                                                       | Проверить меню и якорные переходы.                                  |
| Содержательные изображения имеют осмысленный `alt`, декоративные — пустой `alt` или скрыты от screen reader. | Проверить разметку.                                                 |
| Важная информация доступна без hover.                                                                        | Проверить на touch-логике и без курсора.                            |
| Тема переключается с клавиатуры.                                                                             | Проверить `Tab` + `Enter` / `Space`.                                |
| При `prefers-reduced-motion: reduce` декоративные анимации убираются или упрощаются.                         | Проверить в DevTools.                                               |
| Интерактивные элементы имеют комфортную область нажатия.                                                     | Проверить на мобильной ширине.                                      |
| Текст остаётся читаемым в обеих темах.                                                                       | Проверить контраст и визуально.                                     |

## 12. Блокеры публикации

Сайт нельзя публиковать, если есть хотя бы один из пунктов:

* сломанная навигация;
* неработающий CTA «Смотреть проекты»;
* выдуманный факт, неверное описание проекта, навыка, достижения или опыта;
* нечитаемый текст в светлой или тёмной теме;
* ошибка `npm run lint`;
* ошибка `npm run typecheck` после настройки TypeScript;
* ошибка `npm run build`.

Заглушки вместо реальных `lint`, `typecheck` или `test` не считаются ошибкой раннего каркаса, но являются блокером публичного релиза.

Следующие вещи не блокируют MVP-релиз:

* отсутствие финальных иллюстраций;
* отсутствие готовых обложек всех проектов;
* отсутствие части внешних ссылок;
* отсутствие PDF-резюме по активной ссылке;
* видимые в проектной документации `TODO`;
* отсутствие analytics, SEO и социальных превью;
* отсутствие screenshot-тестов.

Но нельзя публиковать кнопку, которая обещает действие и ведёт на несуществующий или фиктивный адрес.
