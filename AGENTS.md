# AGENTS.md

Инструкции для ИИ-агентов, работающих над личным сайтом-портфолио Кирилла Арзамасцева.

Этот файл обязателен к прочтению перед любыми изменениями в проекте. Его задача - удерживать агентов в рамках уже согласованных продуктовых, контентных, визуальных и технических решений.

## 1. Коротко о проекте

Проект - статический личный сайт-визитка и портфолио Кирилла Арзамасцева.

Цель первой версии: сделать сильное, работающее MVP-портфолио на русском языке, которое быстро объясняет, кто такой Кирилл, показывает реальные проекты, подтверждает сочетание разработки, product/project management и командной работы, а также дает понятный путь к GitHub, Telegram и резюме.

Сайт не должен выглядеть как шаблонное резюме, корпоративный лендинг без характера или полноценная браузерная игра. Основной визуальный образ - аккуратный технологичный сайт с элементом пиксельного офиса в духе PM Simulator.

## 2. Источники правды

Перед началом работы прочитай релевантные документы:

1. `README.md` - обзор проекта, структура, базовые команды.
2. `docs/brief.md` - назначение, аудитории, позиционирование, границы первой версии.
3. `docs/content.md` - единственный источник фактов о Кирилле, проектах, ссылках, навыках, достижениях и контактах.
4. `docs/design-spec.md` - визуальная система, UX, структура страниц, ограничения по дизайну.
5. `docs/acceptance-criteria.md` - Definition of Done, критерии MVP и блокеры публикации.
6. `docs/decisions.md` - журнал принятых решений.

Если документы конфликтуют, используй такой приоритет:

1. `docs/content.md` для фактов и публичного контента.
2. `docs/decisions.md` для принятых продуктовых, технических и процессных решений.
3. `docs/acceptance-criteria.md` для готовности и блокеров.
4. `docs/design-spec.md` для визуала и UX.
5. `docs/brief.md` для общего смысла и позиционирования.
6. `README.md` для справочной информации.

Если конфликт нельзя разрешить безопасно, не выдумывай решение. Зафиксируй вопрос в отчете и выбери минимальный обратимый вариант только там, где это не меняет стратегию проекта.

## 3. Технический стек

Используй уже принятое решение:

- Vite;
- React;
- TypeScript;
- npm и `package-lock.json`;
- React Router для маршрутов `/` и `/employer`;
- CSS Modules для компонентных стилей;
- глобальные CSS-файлы для дизайн-токенов, тем, типографики и базовых правил.

Не добавляй без отдельного согласования:

- backend;
- базу данных;
- CMS;
- авторизацию;
- API;
- серверную форму обратной связи;
- отправку писем с сайта;
- аналитику;
- Tailwind CSS;
- UI-kit;
- state manager;
- отдельную библиотеку анимаций;
- Playwright или visual regression tests до первого релиза.

## 4. Ожидаемая структура проекта

Ориентируйся на такую структуру:

```text
src/
  app/          # инициализация приложения, маршрутизация, layout
  components/   # переиспользуемые UI-компоненты
  sections/     # крупные секции страниц
  data/         # структурированные данные сайта
  styles/       # глобальные стили, темы, токены
  assets/       # локальные ассеты приложения
public/
  images/       # публичные изображения
  resume/       # PDF-резюме
tests/          # тесты, если появляются
docs/           # проектная документация
```

Для контента соблюдай поток:

```text
docs/content.md -> src/data/* -> React-компоненты
```

Не вшивай биографические тексты, ссылки и факты прямо в JSX, если они должны жить в структурированных данных.

## 5. Рабочий процесс агента

Перед изменениями:

1. Проверь текущее состояние репозитория командой `git status -sb`.
2. Прочитай документы, которые относятся к задаче.
3. Проверь существующие файлы и паттерны реализации.
4. Не трогай чужие несвязанные изменения.

Во время работы:

- меняй только файлы, необходимые для задачи;
- предпочитай минимальные, обратимые решения;
- не переписывай архитектуру без причины;
- не меняй `docs/decisions.md` самостоятельно;
- не добавляй новые стратегические решения без явного подтверждения Кирилла;
- если принимаешь локальное предположение, укажи его в итоговом отчете.

После работы:

- запусти релевантные проверки;
- кратко перечисли измененные файлы;
- укажи, какие проверки прошли, а какие не запускались и почему;
- явно назови оставшиеся риски или TODO, если они есть.

## 6. Правила контента

Все публичные факты бери только из `docs/content.md`.

Запрещено:

- придумывать опыт работы;
- придумывать количественные результаты;
- добавлять технологии, которые не подтверждены документацией;
- добавлять фиктивные URL;
- публиковать данные из разделов "Не выводить публично";
- называть Кирилла senior-специалистом, экспертом или опытным product manager без подтверждения;
- обещать результат, которого нет в документации;
- маскировать отсутствующую ссылку под рабочую кнопку.

Если ссылка, PDF, изображение, видео или сертификат еще не добавлены:

- в коде используй безопасное состояние без перехода на фиктивный URL;
- в документации можно оставить явный `TODO`;
- на публичной странице не показывай `TODO` как пользовательский контент.

Основные CTA первой версии:

- "Смотреть проекты";
- "Открыть GitHub";
- "Скачать резюме";
- "Написать в Telegram";
- "Для работодателя".

Не используй расплывчатые подписи вроде "Подробнее", "Узнать больше", "Перейти", если действие можно назвать конкретно.

## 7. Дизайн и UX

Соблюдай `docs/design-spec.md`.

Ключевые принципы:

- mobile-first;
- главная цель первого экрана - быстро объяснить позиционирование и привести к проектам;
- проекты важнее декоративных эффектов;
- пиксельный офис - визуальная фишка, а не полноценная игра;
- светлая и темная темы обязательны для MVP;
- интерфейс должен быть понятным на ширинах 390, 768, 1024 и 1440 px;
- текст и CTA должны оставаться читаемыми поверх любых визуальных элементов;
- анимации короткие, спокойные и отключаются через `prefers-reduced-motion`;
- важная информация не должна быть доступна только по hover.

Не делай:

- полноценную игру на главной странице;
- тяжелую 3D-сцену;
- сложный параллакс;
- fake dashboard;
- перегруженные градиенты;
- стеклянные карточки во всех секциях;
- progress bar или рейтинги навыков;
- длинную биографическую стену текста;
- отдельные страницы для каждого проекта в первой версии;
- блог, форму обратной связи, личный кабинет или комментарии.

## 8. Доступность и качество интерфейса

Каждая пользовательская часть должна учитывать:

- семантическую HTML-разметку;
- логичный порядок заголовков;
- управление клавиатурой;
- видимый focus-state;
- достаточный контраст в обеих темах;
- осмысленный `alt` для содержательных изображений;
- пустой `alt` или скрытие от screen reader для декоративных изображений;
- комфортные области нажатия на мобильных устройствах;
- отсутствие горизонтального скролла на 390 px.

Перед завершением UI-задачи проверь вручную:

- `/`;
- `/employer`, если маршрут затронут;
- светлую и темную тему;
- мобильное меню, если оно есть;
- CTA "Смотреть проекты";
- прямое открытие `/employer`.

## 9. Проверки

Базовые команды:

```bash
npm run lint
npm run typecheck
npm run build
```

Если есть тесты или добавлена интерактивная логика:

```bash
npm run test
```

Текущие `lint`, `typecheck` и `test` могут быть временными заглушками. Если команда является заглушкой, обязательно напиши об этом в отчете. Перед публичным релизом `lint`, `typecheck` и `build` должны быть реальными проверками.

Для UI-изменений одной сборки недостаточно: нужен ручной просмотр в браузере или явное указание, что он не выполнялся.

## 10. Git и безопасность

Не выполняй destructive-команды без прямого запроса Кирилла.

Запрещено:

- `git reset --hard`;
- `git checkout --` для отката чужих изменений;
- удаление файлов вне явной задачи;
- коммит секретов, `.env`, токенов, паролей, SSH-ключей;
- публикация email, города и других непубличных персональных данных;
- изменение DNS, домена, хостинга или deployment secrets.

Перед коммитом или передачей результата проверь:

- `git status -sb`;
- `git diff`;
- что нет случайных изменений в `.codex/`, локальных конфигурациях и секретах.

Если в рабочем дереве есть изменения, которые агент не делал, не трогай их и не откатывай. Если они мешают задаче, остановись и сообщи, что именно конфликтует.

## 11. Решения и документация

`docs/decisions.md` ведется как журнал ключевых решений. Агент читает его перед значимыми изменениями, но не редактирует самостоятельно.

Новое решение требует согласования, если оно меняет:

- стек;
- архитектуру;
- структуру маршрутов;
- публичный контент;
- визуальное направление;
- безопасность;
- деплой;
- CI;
- правила публикации.

Для мелких локальных решений не нужно обновлять `docs/decisions.md`. Достаточно описать предположение в итоговом отчете.

## 12. Деплой

Агент не выполняет деплой.

Агент может:

- подготовить production build;
- проверить `npm run build`;
- написать инструкцию для ручной публикации;
- проверить готовность проекта к ручному деплою.

Агенту запрещено:

- публиковать сайт;
- создавать production release;
- подключать домен;
- менять DNS;
- заходить в панели хостинга;
- создавать deployment workflow;
- добавлять deployment secrets;
- запускать команды, которые выгружают сайт во внешнюю среду.

## 13. CI

GitHub Actions разрешены только для проверок качества.

Разрешенные шаги:

```text
npm ci
npm run lint
npm run typecheck
npm run test
npm run build
```

CI не должен публиковать сайт, создавать release, менять DNS, использовать deployment secrets или загружать production-артефакты на внешний хостинг.

## 14. Поведение при неоднозначности

Если задача не полностью описана:

1. Проверь документацию.
2. Проверь существующий код.
3. Выбери минимальный обратимый вариант, если это не меняет продуктовую стратегию.
4. Не добавляй новую зависимость без явной пользы.
5. Не меняй публичные факты без `docs/content.md`.
6. Сообщи о предположении в итоговом отчете.

Спрашивай Кирилла, если решение влияет на продукт, дизайн-направление, публичные факты, безопасность, домен, деплой или архитектуру.

## 15. Definition of Done для агента

Задача считается завершенной, когда:

- результат соответствует исходной задаче;
- изменения не выходят за нужный scope;
- контент не противоречит `docs/content.md`;
- дизайн не противоречит `docs/design-spec.md`;
- принятые решения из `docs/decisions.md` не нарушены;
- релевантные проверки запущены;
- UI-изменения просмотрены хотя бы на ключевых сценариях или явно указано, что просмотр не выполнялся;
- в отчете есть измененные файлы, проверки и известные ограничения.
